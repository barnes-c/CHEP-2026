\subsection{Provisioning Workflow}

Provisioning follows a CI/CD-driven workflow in which infrastructure intent is translated into provisioning actions and CM inputs. An operator commits changes to the infrastructure repository. The CI/CD pipeline executes validation and a plan operation, producing an auditable view of intended changes. If approved, the pipeline applies the changes: Terraform/OpenTofu provisions cloud resources and registers the host with supporting services through custom providers. The instance boots, cloud-init executes the bootstrap logic, and the Puppet agent performs its first catalog request. From this point, configuration convergence is managed by CM and is outside the framework's control. Figure~\ref{fig:provisioning-seq} illustrates this workflow.

The key design property is that provisioning and CM inputs are derived from a single declarative definition, reducing manual coordination at the provisioning/CM boundary. Updates follow the same plan/apply cycle. The framework does not implement continuous reconciliation; drift is detected through plan output and corrected by re-applying the declared state.
